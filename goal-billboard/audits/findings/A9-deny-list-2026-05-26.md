# A9 — Deny-list audit (pattern lens — overlap, obsolescence, coverage gaps)

Companion to `deny-list-audit-2026-05-26-phase1.md` which produced the per-entry verdict (catastrophic / load-bearing / defensive / friction). This pass looks at the deny list as a **whole** — patterns across entries, not entries in isolation. Three questions:

1. Are any rules **redundant** with stricter rules elsewhere?
2. Are any rules **obsolete** — protecting against operations the agent has no path to execute?
3. Is there **missing coverage** — destructive operations the deny list doesn't catch?

68 deny entries reviewed. **0 settings.json changes proposed; advisory only.**

## 1. Overlap / redundancy

### 1.1 `rm -r*` already covers `rm -rf*`

```
Bash(rm -rf*)    ← matches "rm -rf foo"
Bash(rm -r*)     ← matches "rm -rf foo" AND "rm -rdir foo"
Bash(rm -f*)     ← matches "rm -foo" but ALSO "rm -fr foo"
```

The `rm -r*` glob matches **anything starting with `rm -r`**, including `rm -rf foo`. The `rm -rf*` line is structurally subsumed by `rm -r*`.

**Verdict: KEEP. Redundancy here is a feature.** Defense-in-depth — if the matcher engine ever changed glob semantics, the explicit `rm -rf*` entry survives. Zero cost.

### 1.2 `git branch -d*` already covers `git branch -d calibration-*` and `git branch -d change-*`

```
Bash(git branch -d*)             ← matches all
Bash(git branch -d calibration-*) ← matches subset
Bash(git branch -d change-*)      ← matches subset
```

The specific-pattern entries are subsumed by the wildcard. But the specific entries serve **documentation value** — they signal "we explicitly considered these patterns and chose to keep them denied." Useful when reviewing the deny list.

**Verdict: KEEP. Documentation-by-listing is worth the duplication.** No agent path executes faster or slower due to multiple matches.

### 1.3 `git remote remove*` and `git remote rm*` are aliases

`git remote rm` is the documented alias of `git remote remove`. Both forms are valid CLI invocations; both are denied.

**Verdict: KEEP both.** Without listing the alias, an agent could literally bypass via the shorter form. Necessary explicit coverage.

### 1.4 `git worktree remove .`, `..`, `../EviumOverhaul`, `../EviumOverhaul/`

Four entries cover the same "self-removal" intent but with different argument shapes. Each shape would otherwise reach the actual git command.

**Verdict: KEEP all four.** Each matches a distinct literal that an agent could plausibly construct. (Agent doing `cd ..; git worktree remove EviumOverhaul/` would not be caught by `git worktree remove ../EviumOverhaul/`, but `git worktree remove EviumOverhaul/` is not on the list either — see Coverage Gap 3.2 below.)

## 2. Obsolescence — possible drop candidates

### 2.1 `Bash(git config --unset*)` — has agent ever needed config edits?

git config edits are rare for our workflow. Reviewing 6 session traces (`~/.claude/projects/.../`) finds **zero** instances of the agent attempting to edit git config. The deny is cheap insurance — but possibly an entry the audit catches when we trim the list.

**Verdict: KEEP. The trigger frequency is zero, but so is the cost.** Drop only if list bloat becomes a real cost (currently 68 entries × ~50 bytes each = ~3 KiB; insignificant).

### 2.2 `Bash(git switch*)` — modern alternative

`git switch` is the modern equivalent of `git checkout` (branch switching). Our agent does not switch branches (worktrees handle that). But denying ensures we catch any future attempt.

**Verdict: KEEP. Cheap insurance against branching workflow drift.**

### 2.3 `Bash(npm uninstall*)` — would break the build

The current agent has no need to uninstall npm packages. The build's package.json is user-controlled. Denying ensures dependency changes go through user review.

**Verdict: KEEP. Load-bearing for build integrity.**

## 3. Missing coverage — potential gaps

### 3.1 `pkill` / `killall` against host processes

The deny list covers file system + git destructive operations but **does not cover process termination** of host services. An agent could plausibly run:

```bash
pkill -9 node          # would kill all node processes including the dev server
killall -9 chrome      # would close user's actual browser
killall -9 Code        # would close user's editor
```

None of these are currently denied. The Bash allow list permits `pgrep`, `ps aux`, but neither `kill` nor `killall` is in the allow list — so without an explicit allow, claude-code's default behavior is "prompt user." Effectively that means denied-by-default, but no explicit deny rule.

**Recommendation: add `Bash(pkill*)` and `Bash(killall*)` to deny.** Defensive — there's no legitimate agent workflow that uses them.

```jsonc
"Bash(pkill*)",
"Bash(killall*)",
```

Risk if removed: agent kills user's editor or browser mid-session. Low frequency but high impact.

### 3.2 `git worktree remove` with bare names (no `../`)

The current denies cover:
- `git worktree remove .`
- `git worktree remove ..`
- `git worktree remove ../EviumOverhaul`
- `git worktree remove ../EviumOverhaul/`
- `git worktree remove /*`
- `git worktree remove ~/*`

But **not** `git worktree remove EviumOverhaul` (bare name; would resolve to current directory's sibling if run from parent). Likely uncommon but a gap worth closing.

**Recommendation: add `Bash(git worktree remove EviumOverhaul*)`.** Catches the bare-name case.

### 3.3 `dd` (low-level disk write)

`dd` is the canonical "destroy disk" command. Not on either list. Agent has no need for it.

**Recommendation: add `Bash(dd*)`** as paranoid coverage.

### 3.4 `chmod 000` / `chmod 700 ~/`

The allow list has `chmod +x*`, `chmod 644*`, `chmod 755*` — narrow scope. But the agent could run `chmod 000 .claude/settings.json` (making it unreadable to itself, breaking subsequent operations) without a deny match — it just wouldn't match the allow list and would prompt.

**Verdict: NO ACTION.** The narrow allow list already covers the only chmod patterns the agent legitimately needs. Anything else prompts the user. Adequate.

### 3.5 macOS-specific destructive operations

```bash
# Not currently denied:
mdutil -E /          # erases Spotlight index
defaults delete ...  # nukes app preferences  
launchctl unload ...
```

**Recommendation: NO ACTION.** Agent never invokes these. Coverage-by-default-prompt is enough.
