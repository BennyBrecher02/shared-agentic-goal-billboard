# Device-capture wrappers' fixed render waits fire before first paint on cold boot — white/black frames reported as successful captures

## Description
Found during the OWED mobile-dashboard matrix verify (2026-07-06, http://localhost:43702/mobile/). 4 of 6 first-pass canon-tier captures were NON-RENDERS presented as success: scripts/ios-simulator-capture.sh (sleep 3 post-boot + sleep 4 post-openurl, ~L130/L144) captured WHITE mid-load Safari on ALL THREE iOS tiers on cold boot (evidence: reports/captures/ios/iPhone-SE-(3rd-generation)-20260706-215239.png, iPhone-17-Pro-Max-20260706-215259.png, iPad-Pro-11-inch-(M5)-20260706-215322.png — URL bar shows loading X); scripts/android-emulator-capture.sh (fixed 20s Chrome wait) captured an ALL-BLACK unpainted frame on the small tier (reports/captures/android/Pixel_4a_API_34-20260706-215245.png). Each printed "✓ Captured" — a matrix consumer would green-light on non-evidence (silent-failure class S5). Re-captures with render-VALIDATED polling (screenshot every 8-10s until mean-luma/content check passes; see the -validated-*.png siblings) succeeded on every tier within 8-10s of polling, so the fix direction is replace fixed sleeps with a paint-validation poll (dark-ground mean-luma < 60 for this page, or a generic DOM/paint probe) + a validation failure exit code. Wrapper-level defect, not a page defect — the /mobile/ page itself renders correctly on all 6 devices once loaded.
